# תוכנית מאוחדת למערכת תמלול (עברית) - Master Plan

## 1) מטרות המערכת
1. מערכת תמלול חזקה, אמינה וסקיילבילית לשיחות טלפון בעברית.
2. דיוק מקסימלי כתנאי עליון, גם כשהחומרה מוגבלת.
3. עבודה רב-חברתית (Multi-tenant) עם בידוד מלא בין חברות.
4. תמיכה בגישה גם דרך ממשק משתמש וגם דרך API.
5. תמלול קבצים מתיקייה, כולל קבצים מרובי ערוצים, עם תור חכם והוגן.

## 2) החלטה אסטרטגית: API או WebSocket
הבחירה המומלצת: `API-first + WebSocket`.

1. `REST API` לפעולות עסקיות:
- התחברות והרשאות
- פתיחת משימות תמלול
- חיפוש ושליפת תמלולים
- הורדות (SRT/TXT/DOCX)
- סטטיסטיקות וניצול חבילה

2. `WebSocket` לעדכונים חיים:
- התקדמות תמלול בזמן אמת
- אירועי סיום/שגיאה
- אומדן תור
- אירועי נגן ותמלול חי

למה לא WS בלבד: תחזוקה, versioning, הרשאות ואינטגרציות ארגוניות פשוטים ונכונים יותר ב-API.

## 3) ארכיטקטורה מוצעת (High-Level)
1. `API Gateway` (למשל FastAPI/NestJS)
2. `Auth + Tenant Service` (חברות, משתמשים, RBAC, API keys)
3. `Ingestion Service` (סריקת תיקייה, ולידציה, metadata, ערוצים)
4. `Queue + Scheduler` (ניהול תורים, הוגנות, עדיפויות, עומסים)
5. `Transcription Workers` (GPU/CPU auto-detect, ASR, diarization, alignment)
6. `Post-Processing` (פיסוק, נרמול עברית, ייצוא פורמטים)
7. `Data Layer`:
- PostgreSQL לנתונים מובנים
- Object Storage לקבצי אודיו/פלט
- Redis לתורים/locks/cache
8. `Web UI` (דשבורד, חיפוש, נגן, תמלול)
9. `Observability` (metrics/logs/traces/alerts)

## 4) חומרה וניצול משאבים חכם
### 4.1 זיהוי אוטומטי
בעת עליית השירות:
1. זיהוי CPU (ליבות/threads)
2. זיהוי RAM פנוי
3. זיהוי GPU (NVIDIA/ROCm/DirectML), גרסה וזיכרון VRAM
4. בדיקת זמינות ספריות תלויות (CUDA/cuDNN וכו')

### 4.2 בקרת יציבות
1. תקרות גלובליות:
- CPU: לא לעבור סף עבודה (למשל 75%-80%)
- RAM: לשמור headroom קבוע (למשל 20%)
- VRAM: guardrail (למשל לא לעבור 85%)
2. התאמות בזמן אמת:
- הורדת/העלאת concurrency לפי עומס
- fallback בין GPU ל-CPU
- pause intake קצר בזמן חריגה קיצונית
3. מניעת קריסות:
- retries עם backoff
- watchdog לכל worker
- Dead Letter Queue למשימות שנכשלו שוב ושוב

### 4.3 עיקרון דיוק
אם אין מספיק חומרה:
1. לא מורידים מודל באיכות (Accuracy-first)
2. מורידים קצב/מקביליות בלבד
3. אופציה לעומס גבוה: offload ל-worker חזק חיצוני עם אותו סטנדרט דיוק

## 5) עברית - דרישות חובה
כל השיחות הן בעברית. לכן:

1. לקבע שפה ל-`he` בכל בקשת תמלול (לא auto-detect כשפה ראשית).
2. לשמור שני ייצוגים טקסטואליים:
- `raw_text` (פלט מודל)
- `normalized_text` (לחיפוש, אינדוקס, התאמות)
3. נרמול עברית הכרחי:
- הסרת ניקוד/טעמי מקרא
- נרמול גרש/גרשיים
- איחוד וריאציות Unicode
- טיפול עקבי במספרים, תאריכים, מזהים
4. תמיכה מלאה ב-RTL/BiDi ב-UI ובייצוא (SRT/TXT/DOCX), כולל טקסט מעורב עברית/אנגלית/מספרים.
5. שמירת חותמות זמן ברמת מילה (`word timestamps`) לכל מילה.
6. מדדי איכות חובה:
- `WER` וגם `CER` (לא רק WER)

## 7) קבצים, תיקיות וערוצים
1. תמיכה התחלתית: `wav`, `mp3`
2. תכנון plugin לסיומות עתידיות (`flac`, `m4a`, `ogg`, וכו')
3. בחירת תיקייה לסריקה:
- גילוי קבצים חדשים
- מניעת קליטה כפולה (hash/file-id + idempotency)
4. קובץ מרובה ערוצים:
- כל ערוץ הופך לתת-משימה נפרדת
- תמלול ערוץ-ערוץ גם אם לוקח יותר זמן
5. לאחר תמלול:
- איחוד לתצוגת שיחה מלאה עם שיוך ערוץ/דובר

## 8) תור חכם, הוגנות וריבוי חברות
1. תור רב-דיירים (Multi-tenant queue)
2. אלגוריתם הוגנות: `Weighted Fair Queuing`
3. מגבלות לכל חברה:
- quota חבילה
- max concurrent jobs לפי חבילה/רישיון
4. כללים מרכזיים:
- לא לעבור ספי מערכת בשום מצב
- חלוקה הוגנת בין חברות פעילות
- אם יש משאבים: ריבוי תמלולים במקביל
- אם אין: מעבר אוטומטי לסדרתי
5. SLA-aware scheduling:
- עדיפות למשימות קצרות/קריטיות לפי מדיניות
- בלי לפגוע בהוגנות בסיסית

## 9) הפרדת דוברים (Diarization)
1. Multi-channel:
- שיוך דובר לפי ערוץ קודם לכל
2. Mono:
- diarization אקוסטי (למשל pyannote-like pipeline)
3. מיפוי מילים לדובר:
- שיוך כל מילה ל-speaker_id לפי חפיפה זמנים
4. שמירת confidence לסגמנט/דובר

## 15) קריטריוני קבלה (Definition of Done)
1. דיוק תמלול עברית עומד ביעד WER/CER שהוגדר.
2. תור רב-חברתי עומד בהוגנות ללא חריגת משאבים.
3. תמיכה מלאה בקבצים מרובי ערוצים.
4. נגן+תמלול מסונכרן ברמת מילה.
5. בידוד מידע מלא בין חברות מאומת בבדיקות.
6. API + WS מתועדים ויציבים לשימוש חיצוני.

## 16) סיכום החלטות ליבה
1. ארכיטקטורה: `API-first + WebSocket`
2. עקרון מוצר: `Accuracy-first` (לא מורידים איכות בגלל חומרה)
3. עברית: שפה מקובעת, נרמול מלא, RTL/BiDi מלא, מדידה כפולה WER+CER
4. תפעול: תורים הוגנים, בקרת עומס קשיחה, observability מלא
5. יישום: שלבים קצרים, מדידים, עם בדיקות איכות מהתחלה
